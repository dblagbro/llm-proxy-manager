What is LLM Proxy Manager?

                    
LLM Proxy Manager is a production-ready API proxy that sits between your AI applications and multiple LLM providers. It presents a single Anthropic-compatible API endpoint while automatically routing requests to the best available provider, handling failover, tracking costs, and giving you full visibility into every conversation.

                    
How it works

                    
    
                        
  1. Your app sends a standard Anthropic POST /v1/messages request to the proxy.
    2. 
                        
  2. The proxy selects the highest-priority enabled provider that can handle the request.
    3. 
                        
  3. It translates the request into that provider's native format (Gemini, OpenAI, etc.) and sends it.
    4. 
                        
  4. The response is translated back into Anthropic format and returned to your app.
    5. 
                        
  5. If a provider fails, the proxy automatically tries the next one — up to 3 passes.
    6.

How Requests Are Routed

                    
Every incoming request goes through a multi-stage pipeline before reaching a provider:

                    
    
                        
  1. Hold-down filter — providers that have recently failed are skipped until their cooldown expires (visible in the Monitoring section).
    2. 
                        
  2. Capability router — providers that can't satisfy the request are excluded:
                            
      
                                
    • Requests with tools → skips providers without tool-calling support (e.g. Ollama)
      • 
                                
    • Requests with image content → skips providers without vision (e.g. Grok, Ollama)
      • 
                                
    • Very long context → skips providers with insufficient context windows
      • 
                            
    
                        
    3. 
                        
  3. 3-pass loop — tries each remaining provider in priority order, up to 3 full passes:
                            
      
                                
    • Turn validator (Gemini only) — warns if the message structure is invalid (consecutive same-role turns, empty parts)
      • 
                                
    • Latency guard — if the provider exceeds Max Latency ms, it fails over. Default: 1800ms. Set to 0 to disable.
      • 
                                
    • XML sentinel — scans the response for leaked internal XML tags (e.g. <execute_bash>, <thought>). If found, fails over and tries the next provider.
      • 
                            
    
                        
    4. 
                        
  4. If all providers are exhausted, returns 503 overloaded_error.
    5. 
                    

                    
Hold-Down / Circuit Breaker

                    
When a provider fails repeatedly, it enters a hold-down state and is skipped for a cooldown period. At 90% of the timer it gets one retest — if that succeeds it returns to normal. You can manually release a provider from hold-down in the Monitoring section of the main page.

                    
Max Latency Setting

                    
Each provider has a configurable Max Latency (ms) field (default 1800ms). If a provider takes longer than this to respond, the request is immediately failed over to the next provider. Set to 0 to wait indefinitely.

                


                

                    
Per-Provider Chat Logs

                    
Every request routed through a provider is recorded in a human-readable log file at /app/logs/chat-<provider-name>.log. The format shows the full conversation and outcome:

                    
[2026-04-01 14:23:11 UTC] ── REQUEST → My-Provider (pass 1, model: claude-sonnet-4-6) ──
────────────────────────────────────────────────
[SYSTEM]
You are a helpful assistant.
[USER]
What is the capital of France?
────────────────────────────────────────────────

[2026-04-01 14:23:12 UTC] ── RESPONSE ← My-Provider ──
[ASSISTANT]
The capital of France is Paris.
────────────────────────────────────────────────
latency=312ms  model=claude-sonnet-4-6  tokens=in:14/out:8  cost=$0.000163
────────────────────────────────────────────────

[2026-04-01 14:23:15 UTC] ✗ FAILOVER from Other-Provider (pass 1): Request failed (HTTP 529)

Client API Keys

                    
Client API keys let your applications authenticate with the proxy without exposing the underlying provider API keys. The proxy validates the key before forwarding the request.

                    
Creating a Key

                    
    
                        
  1. Scroll to the Client API Keys section on the main page.
    2. 
                        
  2. Click Generate New API Key.
    3. 
                        
  3. Enter a descriptive name (e.g. "My Python App").
    4. 
                        
  4. Click Generate Key. Copy the key immediately — it is shown only once.
    5. 
                    

                    
Using a Key

                    
Pass the key in the x-api-key header or as a Bearer token:

                    
curl -X POST https://www.voipguru.org/llmProxy/v1/messages \
  -H "x-api-key: llm-proxy-YOUR-KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"claude-sonnet-4-6","max_tokens":1024,"messages":[{"role":"user","content":"Hi"}]}'

                    
Or with the Anthropic SDK:

                    
export ANTHROPIC_API_KEY="llm-proxy-YOUR-KEY"
export ANTHROPIC_BASE_URL="https://www.voipguru.org/llmProxy"

                    
Managing Keys

                    
    
                        
  • Keys can be disabled (blocks new requests, preserves stats) or deleted (permanent).
    • 
                        
  • Each key tracks total requests and last used timestamp.
    • 
                        
  • You can have as many keys as you need — one per application is recommended.
    • 
                    

                


                

                    
Settings

                    
Open Settings from the ⚙️ Settings button in the header.

                    
Session Timeout

                    
Sets how long a login session lasts before requiring re-authentication. Default is 480 minutes (8 hours). Set to a lower value for shared/public terminals.

User Management

                    
Scroll to the User Management section on the main page to add, edit, or remove users. Each user has a username, password, and optional email address (required for password reset).

                    
Password Reset

                    
On the login page, click Forgot Password? and enter your username. A reset link valid for 1 hour will be sent to your registered email. Requires SMTP to be configured.

                


                

                    
Cluster / Multi-Node Deployment

                    
LLM Proxy Manager supports running multiple nodes that sync configuration and report health to each other. Each node operates independently — it does not route requests to peers; syncing is config/stats only.

                    
Environment Variables

                    

                        
VariableDescription

                        
CLUSTER_ENABLED=trueEnables cluster mode

                        
CLUSTER_NODE_URLThis node's public URL (e.g. https://www.voipguru.org/llmProxy)

                        
SESSION_SECRETSet to a stable value so sessions survive container restarts

                        
USE_SQLITE=trueUse SQLite for persistent stats (recommended)

                    

                    
Node 2 — Behind nginx container

                    
If Node 2 is reverse-proxied by a containerised nginx, add the container to the nginx network with the expected alias:

                    
docker run ... \
  --network llm-proxy_llm-proxy-network \
  --network-alias llm-proxy \
  ...

                    
Adding a Cluster Node via UI

                    
    
                        
  1. Scroll to the Cluster section on the main page.
    2. 
                        
  2. Click Add Node and enter the node's URL.
    3. 
                        
  3. The node appears with its last heartbeat time and sync status.
    4. 
                    

                


                

                    
Pointing Claude Code CLI at the Proxy

                    
Claude Code CLI and any application using the Anthropic SDK can be pointed at the proxy by setting two environment variables:

                    
# Use your client API key (or any value if no keys are configured)
export ANTHROPIC_API_KEY="llm-proxy-YOUR-CLIENT-KEY"
export ANTHROPIC_BASE_URL="https://www.voipguru.org/llmProxy"

# Then use Claude Code as normal
cc "summarise this file" -- myfile.txt

                    
For local development:

                    
export ANTHROPIC_BASE_URL="http://localhost:3000"

                    
Python / SDK Example

                    
import anthropic

client = anthropic.Anthropic(
    api_key="llm-proxy-YOUR-CLIENT-KEY",
    base_url="https://www.voipguru.org/llmProxy"
)

message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello!"}]
)
print(message.content[0].text)

                    
cURL Example

                    
curl -X POST https://www.voipguru.org/llmProxy/v1/messages \
  -H "Content-Type: application/json" \
  -H "x-api-key: llm-proxy-YOUR-CLIENT-KEY" \
  -d '{
    "model": "claude-sonnet-4-6",
    "max_tokens": 1024,
    "messages": [{"role": "user", "content": "Hello!"}]
  }'

                    
Streaming

                    
Streaming works automatically when "stream": true is set in the request body. All providers support streaming where available.

                    
Model Names

                    
You can use any model name in the request — the proxy routes based on provider type, not model name. Specify any valid model for the provider you want to use, or just use the model name your app normally uses and let the proxy figure out the best match.
